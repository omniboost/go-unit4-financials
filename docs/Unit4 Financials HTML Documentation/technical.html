Technical Reference

Web Services URL formats

The web services index page lists all the XMLi services and presents generated links to the corresponding XMLi calls and web service WSDL definitions.

Web Services WSDL URLs

The generated Web Services URLs are composed as follows:

{protocol}://{hostname}/{webapp}/wsdl/{product}/{namespace}/{service}-{version}.wsdl

For example:

http://yourserver/codaws/wsdl/finance/input/input-2.0.wsdl

{protocol}://{hostname}
The name of your Web server, for example http://yourserver.
{webapp}
The name of your web services instance, for example codaws.
{product}
The installed Coda web product supplying web services. This can be assets, common, finance, pim, pop, or billing.
{namespace}
The namespace, for example elementmaster or currency.
{service}
The name of the service, for example input.
{version}
The version of the service, for example 3.0.

XMLi schema URLs

The generated XMLi schema URLs are composed as follows:

{protocol}://{hostname}/{webapp}/wsdl/{product}/{namespace}/{namespace}.xsd

For example:

http://yourserver/codaws/wsdl/finance/input/input.xsd

{protocol}://{hostname}
The name of your Web server, for example http://yourserver.
{webapp}
The name of your web services instance, for example codaws.
{product}
The installed Coda web product supplying web services. This can be assets, common, finance, pim, pop, or billing.
{namespace}
The namespace, for example elementmaster or currency.
{namespace}
The name of the service, for example input.

Optional style attributes for Web Services calls

Certain environments may require a different style of Web service call. To accommodate this, you can add a style attribute to the WSDL URL:

{protocol}://{hostname}/{webapp}/wsdl/{product}/{namespace}/{service}-{version}-{style}.wsdl

For example:

http://yourserver/codaws/wsdl/finance/currency/currency-3.0-(wrapped).wsdl

The following optional style attributes can be added to the generated WSDL URL:

No style (Default)

Generates ws-i compatible WSDLs.

Example:

http://yourserver/codaws/wsdl/finance/currency/currency-3.0.wsdl

(wrapped)

Optionally, add this style attribute when you are calling Web Services in a .Net environment. Setting this option changes the method signature of the generated proxy classes to be more like native .Net services.

Example:

http://yourserver/codaws/wsdl/finance/currency/currency-3.0-(wrapped).wsdl

(bare)
Add this style attribute if you prefer to use multi-part Web service requests rather than the default single part style.

Example:

http://yourserver/codaws/wsdl/finance/currency/currency-3.0-(bare).wsdl

(bare-useschematypes)

Add this style attribute when you are calling the browseledger or input Web Services from a Microsoft InfoPath form.

Example:

http://yourserver/codaws/wsdl/finance/currency/currency-3.0-(bare-useschematypes).wsdl

Use of SOAP Headers

Web Services defines an <Options> element for use within the SOAP Header part of aWeb Service request. This element is described in each WSDL document, the following shows the XML Schema used to describe this element for the ElementMaster service.

<xsd:element name="Options">
  <xsd:complexType>
    <xsd:attribute name="locale" type="xsd:string" />
    <xsd:attribute name="bulkload" type="xsd:boolean" />
    <xsd:attribute name="logtitle" type="com:typeLogTitle" />
  </xsd:complexType>
</xsd:element>

The locale attribute accepts ISO 639 two letter language codes. A list is available at ftp://wwwold.dkuug.dk/isp/iso639.txt.

For information on sending SOAP Headers please consult the technical documentation of your chosen Web Service technology or product. Note that Microsoft InfoPath SP1 does not presently support SOAP Headers by default, though it is possible if InfoPath scripting is used.

How the locale is determined

Generated messages such as error messages appear in the language specified by the active client locale.

Usually, the locale is determined accurately. This is the case with Microsoft InfoPath and.Net applications, for example.

Otherwise the active locale is determined from the SOAP message envelope settings. See Use of SOAP Headers. If this is not available, the locale defaults to that specified in the Accept-Language field of the HTTP message header.
